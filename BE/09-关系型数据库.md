# 关系型数据库

- 数据库是服务端“记东西”的地方。前面说过服务端要无状态，状态就落在这里。
- 关系型数据库（MySQL、PostgreSQL）是绝大多数业务的默认选择，先把它吃透。

## 为什么默认选关系型

- 数据有清晰结构、字段固定、彼此有关系（用户、订单、特效、分类）。
- 需要事务保证一致性（要么都成、要么都不成）。
- 需要灵活查询（按各种条件筛选、关联、聚合）。
- 这三点关系型最擅长，所以是默认起点。不确定用什么时，先上关系型。

## 表、行、列与关系

- 表 ≈ 一类实体（effects 表、users 表）。
- 行 ≈ 一条记录。列 ≈ 一个字段。
- 主键（primary key）：每行的唯一标识（通常是 id）。
- 外键（foreign key）：一张表引用另一张表的主键，表达关系（effect.author_id → users.id）。

```sql
CREATE TABLE effects (
    id          BIGSERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    category    VARCHAR(50)  NOT NULL,
    author_id   BIGINT       REFERENCES users(id),  -- 外键，指向 users.id
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

## SQL 四件套（CRUD）

```sql
-- 查：按条件筛选 + 排序 + 分页
SELECT id, name FROM effects
WHERE category = 'transition'
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- 增
INSERT INTO effects (name, category, author_id) VALUES ('溶解转场', 'transition', 1);

-- 改
UPDATE effects SET name = '柔光溶解' WHERE id = 123;

-- 删
DELETE FROM effects WHERE id = 123;
```

- 关联查询（JOIN）：把有外键关系的表拼起来查。

```sql
SELECT e.name, u.name AS author
FROM effects e
JOIN users u ON u.id = e.author_id
WHERE e.category = 'transition';
```

## 索引：查询快慢的关键

- 索引像书的目录：没索引时查某条件要逐行扫全表（慢）；有索引能直接定位（快）。
- 经验：
    - 经常出现在 `WHERE`、`JOIN`、`ORDER BY` 里的列，考虑加索引。
    - 索引不是越多越好——它会让写入变慢（每次写都要更新索引）、占空间。
    - 主键自带索引；外键列通常也该建索引。

```sql
CREATE INDEX idx_effects_category ON effects(category);
-- 多列联合索引：按 (category, created_at) 一起查/排序时高效
CREATE INDEX idx_effects_cat_time ON effects(category, created_at DESC);
```

- 排查慢查询：用 `EXPLAIN` 看执行计划，确认是否走了索引、有没有全表扫描。

## 事务与 ACID

- 事务：把多步操作打包成一个原子单元，要么全成功，要么全回滚。
- ACID：
    - 原子性（Atomicity）：全成或全不成。
    - 一致性（Consistency）：事务前后数据满足约束。
    - 隔离性（Isolation）：并发事务互不干扰（程度由隔离级别决定）。
    - 持久性（Durability）：提交后落盘，断电不丢。

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;  -- 两条要么都生效，要么 ROLLBACK 都不生效
```

- 类比：转账扣款和到账必须同时成立，绝不能扣了没到。这正是关系型数据库不可替代的能力。

## 隔离级别（知道有这回事）

- 并发事务会有脏读、不可重复读、幻读等问题。隔离级别越高越安全但越慢。
- 常用默认：MySQL 是可重复读（Repeatable Read），PostgreSQL 是读已提交（Read Committed）。
- 高并发改同一行数据时，要了解锁和乐观锁/悲观锁（用版本号字段做乐观锁很常见）。

## 连接池：不要每次请求都新建连接

- 建数据库连接很贵（TCP + 认证）。连接池预先建好一批连接复用，请求来了借一个、用完还回去。
- Java 用 HikariCP（Spring Boot 默认），Python 用框架/驱动自带的池。
- 池大小要配置合理：太小并发上不去，太大压垮数据库。

## 数据库迁移：schema 也要版本化

- 表结构会随业务变化，不能靠“上线时手动执行 SQL”。
- 用迁移工具管理每次 schema 变化：Java 常用 Flyway/Liquibase，Python 常用 Alembic。
- 每个迁移文件只描述一次变更，随代码一起提交、进 CI、部署时自动执行，方便追溯“哪次上线改了表”。
- 线上安全原则：向后兼容、分步发布。
    - 先加可空字段或有默认值的字段，让新旧代码都能跑。
    - 新代码开始写新字段，确认稳定。
    - 等所有旧代码下线且无引用，再删旧字段或收紧约束。
- 不要把“改代码 + 改字段名/删字段”放在一次发布里，滚动发布时新旧代码共存，很容易炸。

## ORM：对象和表的映射

- ORM 把数据库的行映射成代码里的对象，让你少写 SQL。
- Java：JPA/Hibernate（全自动 ORM）、MyBatis（SQL 显式可控）、JdbcTemplate（薄封装）。
- Python：SQLAlchemy、Django ORM。
- 权衡：
    - ORM 省事、防 SQL 注入、可移植；但容易写出低效查询（如 N+1：循环里每次查一次库）。
    - 复杂查询、性能敏感处，宁可手写 SQL。
- 不论用不用 ORM，都要懂底层 SQL，否则出了性能问题不会查。

## 选型建议

- MySQL：生态广、运维成熟、读多写多的通用业务。
- PostgreSQL：功能更强（JSON 字段、地理、复杂查询、更严谨的标准支持），现在新项目很多首选。
- 两者都够用；团队熟哪个用哪个。需要 JSON 灵活字段、复杂分析倾向 PostgreSQL。

## 防 SQL 注入（安全红线）

- 永远不要把用户输入拼进 SQL 字符串。用参数化查询（占位符）。

```python
# 危险：字符串拼接，用户传 "1; DROP TABLE effects" 就完了
cur.execute(f"SELECT * FROM effects WHERE id = {user_input}")
# 安全：参数化，驱动负责转义
cur.execute("SELECT * FROM effects WHERE id = %s", (user_input,))
```

## 小结

- 结构化、要关系、要事务 → 默认关系型数据库。
- 索引决定查询性能，按 WHERE/JOIN/ORDER BY 建，但别滥用。
- 事务保证“要么全成要么全不成”，是关系型的核心价值。
- schema 变化用迁移工具版本化，并按向后兼容方式分步发布。
- 用连接池复用连接，ORM 省事但要懂 SQL，参数化查询防注入。
