# 本地开发与日常工作流

- 这一篇不是新概念，而是把后端日常怎么跑、怎么调、怎么验证串起来。
- 对你这种从 native/工具链转过来的人，最重要的是先建立“服务不是一个单独进程，而是一组依赖一起跑”的直觉。

## 本地工具箱

- JDK 21 + Maven/Gradle：跑 Java/Spring Boot 示例和真实服务。
- Python 3.11+ + venv：跑 FastAPI、AIGC 编排、Slack agent 这类 Python 服务。
- Docker + Docker Compose：本地起 Redis、PostgreSQL、MinIO、Nginx 等依赖服务。
- curl 或 HTTPie：不用写客户端也能直接调 API。
- jq：把 JSON 输出格式化，排查接口返回很方便。
- 数据库客户端：psql、DBeaver、DataGrip 都可以，用来查表、看索引、验证数据。

## 一个后端服务本地跑起来通常需要什么

- 应用进程：你写的 Spring Boot / FastAPI 服务。
- 配置：端口、数据库地址、Redis 地址、下游服务地址、密钥。开发环境可以用 `.env`，但真实密钥不要提交。
- 依赖服务：数据库、Redis、对象存储、消息队列，常用 Docker Compose 一次拉起。
- 数据库迁移：启动前要把表结构迁到当前版本，不能靠“手动执行一堆 SQL”。
- 初始数据：本地调试常需要 seed data，比如几个特效、一个测试用户、一个测试 token。

## 推荐的本地开发循环

- 先看 OpenAPI 或 README，确认接口路径、请求体、状态码。
- 起依赖服务：`docker compose up -d`。
- 跑迁移和测试：确认数据库结构、核心逻辑没坏。
- 启动应用：Java 用 `mvn spring-boot:run`，Python 用 `uvicorn main:app --reload`。
- 用 curl 调接口，看状态码、响应体、日志里的 traceId。
- 改代码后重复：测试 → 启动/热更新 → curl 验证 → 看日志。

## 调一个请求时看什么

- 请求：方法、URL、Header、Body 是否符合契约。
- 响应：状态码是否归因正确，错误体是否带 `code/message/traceId`。
- 日志：入口日志、下游调用日志、异常堆栈是否能用同一个 traceId 串起来。
- 数据：数据库是否真的写入，Redis key 是否按预期变化。
- 依赖：如果请求慢，先看是应用慢、数据库慢、Redis 慢，还是下游 API 慢。

## 从这套教程怎么动手

- 先跑 `examples/05-spring-boot-rest` 和 `examples/06-fastapi-rest`，对照两种框架的同一组接口。
- 再跑 `examples/14-gateway-cluster`，亲眼看请求被 Nginx 分到不同实例、状态却在 Redis 里连续增长。
- 最后跑三个实战示例：20 看签名 URL 下发，21 看 taskId + SSE 进度，22 看 Slack 事件 ACK + 后台处理。

## 小结

- 后端本地开发不是只跑一个进程，而是应用、配置、依赖服务、迁移、测试、日志一起协作。
- curl + OpenAPI + 日志 + 数据库客户端，是最基础也最有效的调试组合。
- 先能稳定本地跑通，再谈部署、集群和线上排障。
