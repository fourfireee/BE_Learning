# 配置与环境管理

- 同一份代码要跑在本地、测试、预发、线上等不同环境，连的数据库、密钥、开关都不同。配置管理就是解决这个。
- 核心原则一句话：代码不变，配置随环境注入。

## 为什么不能把配置写死在代码里

- 写死就意味着换环境要改代码、重新打包，容易出错（拿测试配置发上线）。
- 密钥写进代码会随仓库泄露，是重大安全事故。
- 目标：同一个构建产物（jar/镜像）原封不动地部署到任何环境，靠外部配置切换行为。这是云原生的基本要求（十二要素应用原则）。

## 配置从哪来（优先级从低到高）

```mermaid
flowchart LR
    A[默认值/配置文件] --> B[环境特定配置文件]
    B --> C[环境变量]
    C --> D[启动参数/配置中心]
```

- 配置文件：放默认值和非敏感配置（端口、超时、日志级别）。
- 环境特定文件：`application-dev.yml` / `application-prod.yml`，用 profile 选。
- 环境变量：注入环境相关、尤其是敏感值（数据库密码、密钥）。容器部署最常用这种。
- 配置中心/启动参数：运行时可热更新的配置（开关、限流阈值）。
- 一般后者覆盖前者，这样能在不改文件的情况下临时覆盖某项。

## 区分配置和密钥

- 普通配置：数据库地址、端口、第三方服务 URL、功能开关、超时阈值。可以放配置文件/配置中心。
- 密钥（secrets）：数据库密码、API key、JWT 签名密钥、第三方 token。绝不进代码仓库，单独管理。

## 密钥怎么管

- 最低要求：用环境变量注入，不写进提交的文件。
- 更规范：用密钥管理服务（Vault、云厂商 KMS/Secrets Manager、K8s Secret），统一存储、加密、轮换、审计访问。
- 本地开发：用 `.env` 文件 + 加入 `.gitignore`，并提供一份 `.env.example`（只有 key 没有真实值）给同事参考。

```bash
# .env（加进 .gitignore，不提交）
DB_PASSWORD=realsecret
JWT_SECRET=anothersecret

# .env.example（提交，给别人看格式，不含真实值）
DB_PASSWORD=
JWT_SECRET=
```

## 在框架里读配置

```yaml
# Spring: application.yml，${ENV_VAR:默认值}
spring:
  datasource:
    url: ${DB_URL:jdbc:postgresql://localhost:5432/app}
    password: ${DB_PASSWORD}   # 没有默认值，必须由环境注入
app:
  ai-service-url: ${AI_SERVICE_URL}
  request-timeout-ms: ${REQUEST_TIMEOUT:30000}
```

```python
# Python: 用 pydantic-settings 从环境变量读，带类型校验
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    db_url: str
    db_password: str
    ai_service_url: str
    request_timeout_ms: int = 30000   # 默认值

settings = Settings()   # 启动时从环境变量读取并校验，缺必填项直接报错
```

## 多环境的典型划分

- local：本地开发，连本地或共享的开发依赖。
- dev/test：联调测试环境。
- staging/预发：尽量和线上一致，上线前最后验证。
- prod/线上：真实用户流量，配置最严、权限最小、密钥最敏感。
- 关键：环境之间数据和密钥要隔离，绝不能用线上密钥连测试，也别拿线上数据随便测。

## 配置变更也要谨慎

- 配置改错和代码改错一样能搞挂线上（改错数据库地址、限流阈值设成 0）。
- 配置变更也应走评审、有记录、可回滚。配置中心的变更要有审计和灰度能力。

## 小结

- 一份构建产物 + 外部注入配置 = 能跑在所有环境。
- 配置分普通配置和密钥；密钥绝不入库，用环境变量/密钥服务注入。
- 框架用 `${ENV:默认}` 形式读取；本地用 `.env`（gitignore）+ `.env.example`。
- 环境间数据/密钥严格隔离；配置变更同样要可控可回滚。
