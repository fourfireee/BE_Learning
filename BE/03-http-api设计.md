# HTTP API 设计

- 服务端对外暴露的“接口”就是 API。设计好坏直接影响前端/调用方体验、可维护性和扩展性。
- 这一篇讲业界最常见的 REST 风格，以及状态码、幂等、版本、分页、错误约定这些“契约”问题。

## REST 的核心思想

- 把系统里的东西抽象成“资源”，用 URL 表示资源，用 HTTP 方法表示对资源的操作。
- 资源用名词（复数），不要在 URL 里放动词。

```text
GET    /effects          列出特效
GET    /effects/123      取单个特效
POST   /effects          新建特效
PUT    /effects/123      整体替换特效 123
PATCH  /effects/123      局部更新特效 123
DELETE /effects/123      删除特效 123
```

- 反例（别这么写）：`/getEffect?id=123`、`/createEffect`、`/deleteEffectById`——把动作塞进 URL，方法全用 GET/POST，就丢了 REST 的统一性。

## HTTP 方法与语义

- GET：读取，不应有副作用（不改数据）。
- POST：新建，或“无法用其他方法表达的动作”。
- PUT：用请求体整体替换资源（没带的字段会被清空）。
- PATCH：只改请求体里给的字段，其余不动。
- DELETE：删除。

## 幂等性（重要且容易忽略）

- 幂等 = 同一个请求发一次和发多次，对服务端的最终效果一样。
- 为什么重要：网络会超时重试，客户端可能重复点击，重复请求很常见。幂等的接口重试是安全的。
- 各方法的幂等性：
    - GET、PUT、DELETE：天然幂等（读不改；整体替换重复做结果一样；删一个已删的还是删了）。
    - POST：默认不幂等（连发两次创建两条）。
- 让 POST 幂等的常见手段：客户端带一个唯一的“幂等键”（idempotency key），服务端发现重复就返回上次结果，不重复执行。下单、支付、提交生成任务都该这么做。
- 实现直觉：用 `Idempotency-Key` 或 `clientRequestId` 作为唯一键，服务端把“这个键对应的处理状态/结果”存起来并设置 TTL；重复请求命中同一个键时，返回同一个任务或同一个结果。
- 对 AIGC 长任务特别重要：客户端重试 `POST /v1/tasks` 时，不应创建两个生成任务，而应拿到同一个 `taskId`。

## 状态码：用对了能少写很多文档

- 2xx 成功：
    - 200 OK：成功并有返回体。
    - 201 Created：新建成功（POST 后）。
    - 202 Accepted：已接受、但还在异步处理（很适合 AIGC 长任务：先收下、给个任务 id）。
    - 204 No Content：成功但无返回体（DELETE 常用）。
- 3xx 重定向：301/302/304（304 见缓存）。
- 4xx 客户端错（怪调用方）：
    - 400 Bad Request：参数不合法。
    - 401 Unauthorized：没认证（没登录/token 无效）。
    - 403 Forbidden：认证了但没权限。
    - 404 Not Found：资源不存在。
    - 409 Conflict：冲突（如重复创建、版本冲突）。
    - 429 Too Many Requests：被限流了。
- 5xx 服务端错（怪自己）：
    - 500 Internal Server Error：代码出异常。
    - 502/503/504：网关/下游不可用/超时。
- 原则：错误归因要准确——客户端传错用 4xx，自己挂了才用 5xx。监控里 5xx 才该报警。

## 统一的错误返回结构

- 别只靠状态码，body 里给出可读的错误信息，方便调用方处理。约定一个固定结构，全站一致：

```json
{
  "code": "EFFECT_NOT_FOUND",
  "message": "特效 123 不存在",
  "traceId": "a1b2c3d4"
}
```

- `code`：机器可判断的错误码（调用方按它做分支）。
- `message`：给人看的描述。
- `traceId`：链路追踪 id，排障时拿它去日志里捞（见可观测篇）。

## 版本化

- 接口上线后会被很多人依赖，改动可能破坏老客户端。所以要版本化，给演进留空间。
- 常见做法：放进 URL 前缀，简单直观：`/v1/effects`、`/v2/effects`。
- 原则：能兼容就别升版本（加字段一般兼容、删字段/改语义不兼容）。

## 分页

- 列表接口不能一次返回全部，要分页。两种主流：
- 基于偏移（offset/limit 或 page/size）：`/effects?page=2&size=20`。简单，但数据量大、翻很后面时慢，且翻页中途有增删会错位。
- 基于游标（cursor）：返回一个 `nextCursor`，下次带上它继续取。适合大数据量和无限滚动，性能稳定。

```json
{
  "items": [ ... ],
  "nextCursor": "eyJpZCI6MTIzfQ==",
  "hasMore": true
}
```

## 字段约定与兼容性

- API 一旦给客户端用，就要把“返回字段”当契约看待。
- 兼容改动：新增可选字段、新增枚举值（调用方能兜底）、放宽输入限制。
- 不兼容改动：删字段、改字段类型、改字段含义、把原来可选变必填、把成功状态码改成错误。
- 常见字段约定：
    - 时间用 ISO 8601 字符串或毫秒时间戳，全系统统一时区。
    - 金额、大小、耗时这类数值明确单位，比如 `sizeBytes`、`elapsedMs`。
    - 枚举要有兜底策略，客户端不能假设永远只有当前几个值。

## OpenAPI：接口契约文档

- OpenAPI（旧称 Swagger）是一份描述你所有接口的标准化文件（路径、参数、返回结构、错误码）。
- 价值：能自动生成接口文档页面、生成各语言的客户端 SDK、做契约测试。
- Spring Boot（springdoc）和 FastAPI 都能根据代码自动生成 OpenAPI，几乎零成本，强烈建议开。

## 设计一个接口的检查清单

- 资源命名是名词、用对了 HTTP 方法了吗？
- 这个接口幂等吗？需要幂等键吗？
- 成功/各种失败分别返回什么状态码？
- 错误结构统一吗？带 traceId 了吗？
- 列表分页了吗？
- 入参校验了吗（必填、范围、长度）？
- 字段命名、时间/单位/枚举约定清楚吗？
- 这次改动是否向后兼容？老客户端会不会挂？
- 要不要鉴权？谁能调？（见鉴权篇）

## 小结

- REST：名词资源 + HTTP 方法表达操作。
- 幂等性决定重试是否安全，写接口先问“能不能安全重试”。
- 状态码要归因准确，错误体结构统一并带 traceId。
- 字段约定、版本化、分页、OpenAPI 是长期可维护的基础设施。
