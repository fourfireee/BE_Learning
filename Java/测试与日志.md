# Java 测试与日志

- 真实后端项目里，测试和日志不是附属品，而是定位问题、保护重构、理解线上行为的基础设施。
- 这篇先建立最小心智模型：怎么写单元测试，为什么不用 `System.out` 当日志。

## 测试目录

- Maven / Gradle 的标准目录：

```txt
src/
  main/java/      # 业务代码
  test/java/      # 测试代码
```

- 测试类通常和被测类包名一致，文件名以 `Test` 结尾。

```txt
src/main/java/com/example/PriceCalculator.java
src/test/java/com/example/PriceCalculatorTest.java
```

## JUnit 5

- Java 后端最常见的单元测试框架是 JUnit 5。
- 一个测试通常按 Arrange / Act / Assert 组织：准备数据、执行行为、断言结果。

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.junit.jupiter.api.Assertions.assertThrows;

class PriceCalculatorTest {
    @Test
    void calculatesTotalPrice() {
        PriceCalculator calc = new PriceCalculator();
        int total = calc.total(10, 3);
        assertEquals(30, total);
    }

    @Test
    void rejectsNegativeCount() {
        PriceCalculator calc = new PriceCalculator();
        assertThrows(IllegalArgumentException.class, () -> calc.total(10, -1));
    }
}
```

- Maven 里会把 `src/test/java` 下的测试交给 `mvn test` 执行。
- 示例依赖版本只是写法示意；真实项目常由 Spring Boot parent 或公司 BOM 统一管理版本。

```xml
<dependency>
  <groupId>org.junit.jupiter</groupId>
  <artifactId>junit-jupiter</artifactId>
  <version>5.10.2</version>
  <scope>test</scope>
</dependency>
```

## 单元测试 vs 集成测试

- 单元测试：只测一个类或一小块逻辑，快、稳定、方便重构。
- 集成测试：连数据库、HTTP、消息队列等外部组件，更接近真实环境，但更慢。
- 入门时先给纯业务逻辑补单元测试，再逐步学 Spring Boot 的集成测试。

## 日志门面

- 示例程序用 `System.out.println` 没问题；真实后端不要靠它做日志。
- 常见做法是代码里使用 SLF4J 这个日志门面，运行时接 Logback、Log4j2 等具体实现。
- 类比图形/引擎里的抽象层：业务代码依赖统一接口，底层日志实现可以替换。

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    private static final Logger log = LoggerFactory.getLogger(UserService.class);

    public void createUser(String name) {
        log.info("create user, name={}", name);
    }
}
```

- `{}` 是占位符，日志框架会延迟格式化，比自己拼字符串更好。

## 日志级别

- `trace`：非常细的调试信息，默认通常不开。
- `debug`：开发/排查问题用。
- `info`：关键业务流程、启动信息、重要状态变化。
- `warn`：可恢复但需要关注的问题。
- `error`：失败、异常、需要处理的问题。

## 实践建议

- 测试命名写清行为：`returnsEmptyWhenUserNotFound` 比 `test1` 有用得多。
- 一个测试尽量只验证一个行为，失败时容易定位。
- 日志要带上下文：请求 ID、用户 ID、订单 ID、关键参数。
- 不要在日志里打印密码、token、身份证号等敏感信息。
- 异常日志要保留堆栈：`log.error("failed to create user", e)`，不要只打 `e.getMessage()`。
