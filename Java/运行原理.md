# Java 运行原理

- 这一篇先回答一个核心问题：你写的 `.java` 文件，到底是怎么变成能跑的程序的。
- 和 C++ 对比着看会很快：C++ 是“源码直接编译成某个平台的机器码”，Java 走的是“先编译成中间字节码，再由 JVM 运行”。

## 和 C++ 的最大区别

- C++：源码 → 编译器（针对某个 CPU/OS）→ 机器码可执行文件 → 操作系统直接运行。换平台要重新编译。
- Java：源码 → 编译成字节码（`.class`）→ JVM 在运行时把字节码翻译/执行。换平台只要装对应平台的 JVM，字节码不用改。
- 一句话：C++ 把“适配平台”这件事放在编译期；Java 把它放在运行期，交给 JVM 处理。这就是 “Write Once, Run Anywhere” 的来源。

## 关键名词

- `JDK`：Java Development Kit，开发用的全套工具（含编译器 `javac`、JVM、标准库）。写代码装这个。
- `JRE`：Java Runtime Environment，只负责运行（含 JVM + 标准库），不含编译器。现在一般直接装 JDK 即可。
- `JVM`：Java Virtual Machine，真正执行字节码的“虚拟机”，可以理解为一个软件实现的 CPU。
- `字节码 (bytecode)`：`.class` 文件里的中间指令，不是机器码，而是 JVM 能看懂的统一指令集。
- `JIT`：Just-In-Time 编译器，运行时把“热点”字节码进一步编译成机器码，提升性能。

## 编译与运行流程

```mermaid
flowchart LR
    A[Hello.java 源码] -->|javac 编译| B[Hello.class 字节码]
    B -->|加载| C[JVM 类加载器]
    C --> D[解释执行字节码]
    D -->|热点代码| E[JIT 编译成机器码]
    E --> F[CPU 执行]
```

- 编译期：`javac` 只做一次，把 `.java` 翻译成跨平台的 `.class` 字节码，不针对具体 CPU。
- 运行期：JVM 启动后先“解释执行”字节码（边读边执行，启动快但慢）；对反复执行的“热点”代码，JIT 把它编译成机器码缓存起来，后续就跑得和 native 接近。
- 这就是为什么 Java 程序“跑一会儿会变快”：JIT 在背后慢慢把热点代码优化成了机器码。

## 上手最小例子

- Java 规定：public 类名必须和文件名一致，所以这个类要放在 `Hello.java` 里。

```java
// 一个最小的可运行 Java 程序
public class Hello {
    // main 是程序入口，签名固定：public static void main(String[] args)
    // String[] args 是命令行参数，相当于 C++ 的 argv
    public static void main(String[] args) {
        // System.out 是标准输出，println 会自动换行
        System.out.println("Hello Java");
    }
}
```

编译并运行：

```bash
# 编译：生成 Hello.class 字节码
javac Hello.java

# 运行：注意这里写类名 Hello，不是 Hello.class，也不是 Hello.java
java Hello
```

- 从 Java 11 起，简单单文件还能跳过显式编译，直接 `java Hello.java` 一步运行（JVM 内部帮你编译）。

## 一次编译、到处运行的代价

- 好处：跨平台、内存安全（数组越界、空指针会抛异常而不是像 C++ 那样未定义行为踩内存）。
- 代价：多了一层 JVM，启动有开销、运行时占内存更多；极致性能场景（比如你熟悉的渲染热循环）通常仍是 C++ 更可控。
- 对后端服务来说，这个取舍很划算：开发效率高、生态成熟、长期运行时 JIT 能把性能拉到够用的水平。

## 给你的类比

- 把字节码想象成你熟悉的“中间表示 / IR”：`javac` 像前端把源码降到统一 IR，JVM + JIT 像后端在目标机器上把 IR 真正跑起来并优化。
- C++ 是“编译期就把 IR 一路降到机器码”，Java 是“把降到机器码这步推迟到运行时按需做”。
