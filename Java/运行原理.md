# Java 运行原理

- 这一篇先回答一个核心问题：你写的 `.java` 文件，到底是怎么变成能跑的程序的。
- 和 C++ 对比着看会很快：C++ 是“源码直接编译成某个平台的机器码”，Java 走的是“先编译成中间字节码，再由 JVM 运行”。

## 和 C++ 的最大区别

- C++：源码 → 编译器（针对某个 CPU/OS）→ 机器码可执行文件 → 操作系统直接运行。换平台要重新编译。
- Java：源码 → 编译成字节码（`.class`）→ JVM 在运行时把字节码翻译/执行。换平台只要装对应平台的 JVM，字节码不用改。
- 一句话：C++ 把“适配平台”这件事放在编译期；Java 把它放在运行期，交给 JVM 处理。这就是 “Write Once, Run Anywhere” 的来源。

## 关键名词

- `JDK`：Java Development Kit，开发用的全套工具（含编译器 `javac`、JVM、标准库）。写代码装这个。
- `JRE`：Java Runtime Environment，只负责运行（含 JVM + 标准库），不含编译器。现在一般直接装 JDK 即可。
- `JVM`：Java Virtual Machine，真正执行字节码的“虚拟机”，可以理解为一个软件实现的 CPU。
- `字节码 (bytecode)`：`.class` 文件里的中间指令，不是机器码，而是 JVM 能看懂的统一指令集。
- `JIT`：Just-In-Time 编译器，运行时把“热点”字节码进一步编译成机器码，提升性能。

## 安装 JDK 与多版本管理（SDKMAN）

- 装 JDK 在 macOS 上有几种方式：官网下载安装包、Homebrew、或版本管理器。如果你以后要在多个 Java 版本间切换（不同项目要求不同），一开始就直接用 **SDKMAN** 这条路线最省心——它相当于 Java 界的 `nvm` / `pyenv`，一个工具搞定「下载 + 安装 + 切换」，还能顺带管理 Maven、Gradle、Kotlin。
    - Maven：Java 最主流的构建 + 依赖管理工具，用 XML 文件（`pom.xml`）声明依赖和构建流程，约定大于配置，类比前端的 npm。
    - Gradle：更现代灵活的构建工具，用代码（Groovy/Kotlin 脚本）写构建逻辑，比 Maven 更快更可编程，Android 项目的标准选择。
        - Groovy：一门运行在 JVM 上的动态脚本语言，语法简洁、和 Java 兼容，早期 Gradle 构建脚本默认用它来写（现在也可改用 Kotlin）。
    - Kotlin：运行在 JVM 上的另一门语言，语法更简洁、空安全更好，能和 Java 无缝互调，是目前 Android 开发的首选语言。
    
- 安装 SDKMAN（需要联网）：

```bash
# curl 把官方安装脚本从网上下载下来（-s 是 silent，不显示下载进度条）；
# “|”管道把下载到的脚本内容直接喂给后面的程序去执行，省得先存成文件再运行。
# 为什么是 | bash 而不是 | zsh：这只是“用哪个解释器跑这段安装脚本”，
# 和你平时用的终端无关——脚本是按 bash 语法写的，所以用 bash 执行最稳妥；
# 你的默认 shell 仍然是 zsh，安装完后日常照常用 zsh。
# 注意：安装脚本会自动改你的 shell 配置文件（zsh 是 ~/.zshrc），把 SDKMAN 的初始化加进去，
# 所以重开终端后在 zsh 里就能直接用 sdk 命令。
curl -s "https://get.sdkman.io" | bash
# 装完按提示重开终端，或手动加载（让当前这个窗口立刻能用 sdk，无需重开）：
source ~/.sdkman/bin/sdkman-init.sh
```

- 安装与切换 JDK：

```bash
# 看有哪些可装的版本/厂商（带后缀，如 -tem 是 Temurin，新手选它即可）
sdk list java

# 装多个版本
sdk install java 21.0.3-tem   # 当前 LTS
sdk install java 17.0.11-tem
sdk install java 8.0.412-tem

# 临时切换（只在当前终端窗口生效）
sdk use java 17.0.11-tem

# 设为全局默认
sdk default java 21.0.3-tem

# 查看当前用的是哪个
sdk current java
```

- 按项目自动切换版本（类似 `nvm` 的 `.nvmrc`）：在项目根目录放一个 `.sdkmanrc` 文件，`cd` 进去时自动切到对应版本（需在 `~/.sdkman/etc/config` 里开启 `sdkman_auto_env=true`）。
    - 怎么理解这句括号：`~/.sdkman/etc/config` 是 SDKMAN 自己的配置文件（`~` 是你的用户主目录），里面是一行行 `选项=值` 的开关。
    - `sdkman_auto_env` 就是「进入目录时要不要自动切版本」这个开关，默认是 `false`（关）。把它改成 `true` 后，`cd` 进带 `.sdkmanrc` 的目录才会自动切换；否则就算放了 `.sdkmanrc` 也不生效，得手动 `sdk env` 才切。
    - 一句话：这是个默认关闭的功能，要先去配置文件里把开关打开，自动切换才会工作。

```
java=17.0.11-tem
```

- 小贴士：
    - 默认版本选当前 LTS（21），需要老项目时再装 17 / 8。
    - 苹果芯（M 系列）会自动拿 `arm64` 版本，一般无需关心架构。
    - Maven、Gradle 也用 SDKMAN 装（`sdk install maven`），省得单独配。
    - 如果之前用 Homebrew 装过 JDK，建议统一改用 SDKMAN 管，避免两套管理方式打架、`java -version` 指向混乱。

## 编译与运行流程

```mermaid
flowchart LR
    A[Hello.java 源码] -->|javac 编译| B[Hello.class 字节码]
    B -->|加载| C[JVM 类加载器]
    C --> D[解释执行字节码]
    D -->|热点代码| E[JIT 编译成机器码]
    E --> F[CPU 执行]
```

- 编译期：`javac` 只做一次，把 `.java` 翻译成跨平台的 `.class` 字节码，不针对具体 CPU。
- 运行期：JVM 启动后先“解释执行”字节码（边读边执行，启动快但慢）；对反复执行的“热点”代码，JIT 把它编译成机器码缓存起来，后续就跑得和 native 接近。
- 这就是为什么 Java 程序“跑一会儿会变快”：JIT 在背后慢慢把热点代码优化成了机器码。

## 上手最小例子

- Java 规定：public 类名必须和文件名一致，所以这个类要放在 `Hello.java` 里。

```java
// 一个最小的可运行 Java 程序
public class Hello {
    // main 是程序入口，签名固定：public static void main(String[] args)
    // String[] args 是命令行参数，相当于 C++ 的 argv
    public static void main(String[] args) {
        // System.out 是标准输出，println 会自动换行
        System.out.println("Hello Java");
    }
}
```

编译并运行：

```bash
# 编译：生成 Hello.class 字节码
javac Hello.java

# 运行：注意这里写类名 Hello，不是 Hello.class，也不是 Hello.java
java Hello
```

- 从 Java 11 起，简单单文件还能跳过显式编译，直接 `java Hello.java` 一步运行（JVM 内部帮你编译）。

## 一次编译、到处运行的代价

- 好处：跨平台、内存安全（数组越界、空指针会抛异常而不是像 C++ 那样未定义行为踩内存）。
- 代价：多了一层 JVM，启动有开销、运行时占内存更多；极致性能场景（比如你熟悉的渲染热循环）通常仍是 C++ 更可控。
- 对后端服务来说，这个取舍很划算：开发效率高、生态成熟、长期运行时 JIT 能把性能拉到够用的水平。

## 给你的类比

- 把字节码想象成你熟悉的“中间表示 / IR”（IR = Intermediate Representation，中间表示：介于源码和机器码之间、便于优化和跨平台的统一形式）：`javac` 像前端把源码降到统一 IR，JVM + JIT 像后端在目标机器上把 IR 真正跑起来并优化。
- C++ 是“编译期就把 IR 一路降到机器码”，Java 是“把降到机器码这步推迟到运行时按需做”。
