# Java 面向对象

- Java 是彻底的面向对象语言：所有代码都写在类里，没有游离的全局函数和全局变量。
- 你有 C++ 的 OOP 基础，这一篇重点讲 Java 的差异和约定。

## 类与对象

- `class` 定义一类东西的结构和行为；`new` 创建出来的具体实例就是对象（在堆上）。
- 类的成员：字段（数据）、方法（行为）、构造器（创建时初始化）。

```java
public class User {
    // 字段（成员变量），private 表示只能类内部访问
    private String name;
    private int age;

    // 构造器：名字和类名相同，没有返回类型。new User(...) 时调用
    public User(String name, int age) {
        // this 指向当前对象，用来区分同名的参数和字段
        this.name = name;
        this.age = age;
    }

    // 方法：对外提供的行为
    public String describe() {
        return name + " (" + age + ")";
    }
}
```

```java
// 创建对象一定用 new，返回的是堆上对象的引用
User u = new User("Alice", 18);
System.out.println(u.describe());
// 不需要手动 delete，GC 会在没人引用后自动回收
```

## 封装与访问修饰符

- 习惯做法：字段设为 `private`，通过方法（getter/setter）暴露，外部不能直接乱改内部状态。
- 四种访问级别：
- `public`：所有地方可访问。
- `protected`：同包 + 子类可访问。
    - 「包（package）」是 Java 给类做的分组/命名空间，对应源码的目录结构。「同包」就是指处于同一个 `package` 下的那些类（通常也在同一个目录里）。类比 C++ 的 `namespace`，但 Java 的包同时还参与访问控制。
    - 「用文件顶部的 `package ...;` 声明」是什么意思：每个 `.java` 文件的第一行（注释除外）写一句 `package 包名;`，用来声明「这个文件里的类属于哪个包」。包名一般和它所在的目录路径一一对应。举例：

```text
src/
  com/
    example/
      user/
        User.java        <- 第一行写 package com.example.user;
        UserService.java  <- 同样写 package com.example.user;
```

    - 上面 `User.java` 和 `UserService.java` 都声明了 `package com.example.user;`，所以它俩「同包」，互相能访问对方的 `protected` 和默认（package-private）成员。
    - 包名用「点」分层（`com.example.user`），既是命名空间也是目录层级（`com/example/user/`）。
    - 为什么是 `com.example.user` 而不是 `example.user`：这只是行业惯例，不是语法强制——`example.user` 甚至单个 `user` 都能编译通过。惯例是「把公司域名倒过来写」当包名前缀：域名 `example.com` 倒写成 `com.example`，再接模块名 `user`。
        - 为什么倒写：域名本身就是全球唯一的（`example.com` 全世界只有一个），倒过来后越靠左越「大」（`com` → `example` → `user`），正好和包/目录从外到内的层级顺序一致。
        - 目的：保证你的包名全球不和别人撞。你和别的库都有个 `User` 类时，靠 `com.example.user.User` 和 `org.other.account.User` 这种带前缀的全名区分开——这和 C++ 用 `namespace` 防冲突是一个道理。
        - 所以平时自己练习随便写 `user`、`demo` 也能跑；但正式项目、尤其要发布给别人用的库，一定用域名倒写前缀。
    - 别的文件要用这个类时，就 `import com.example.user.User;`（这也是为什么前面说「没有 #include，用 import 引入其它包的类」）。
    - 不写 `package` 这一行会怎样：代码照样能编译运行（单文件小练习常这么干），此时类属于「默认包 / 无名包（default / unnamed package）」——即没有包名、直接放在源码根目录下的类。但正式项目基本不用，因为：
        - 默认包里的类无法被「有包名」的类 `import`，等于被孤立，别处用不了。
        - 没有命名空间，多人/大项目里极易重名冲突。
        - Maven/Gradle 标准布局和 Spring 组件扫描等默认不照顾默认包，容易出「找不到/扫不到」的怪问题。
            - 「Spring 组件扫描（component scan）」简述：Spring 框架启动时会自动「扫描」指定的包及其子包，把其中标了注解（如 `@Component`、`@Service`、`@Controller`）的类创建成对象（叫 Bean）并管理起来，省得你手动 `new` 和拼装。它从「启动类所在的包」开始往下扫，所以若把类放进默认包（没有包名、在根目录），扫描起点和范围会乱套，扫不到你的类。这块到 Spring/后端教程会展开，这里先知道「不写 package 会破坏它的扫描」即可。
        - 结论：随手练习可以省略，正经代码都应显式写 `package`。
    - 所以 `protected` 的可见范围有两部分：① 同一个包里的其它类；② 任意包里的子类（即使不在同一个包）。
- 默认（不写修饰符）：同包可访问（叫 package-private）。
- `private`：仅本类可访问。

## 继承

- 用 `extends` 继承，只能单继承（一个类只有一个父类），这是和 C++ 的重要区别（C++ 支持多继承）。
- `super` 调用父类构造器或父类方法。
- `@Override` 注解标记“这是在重写父类方法”，写错签名编译器会报错，建议总是加上。

```java
public class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public String speak() { return "..."; }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name); // 必须先调用父类构造器
    }

    // @Override 不是“让它成为重写”的开关，只是一个给编译器看的“断言/检查”。
    // 不写它，这个方法照样是重写（override）：运行时一样会发生多态，调用到 Dog 的版本。
    // 写它的好处：让编译器帮你校验“确实重写到了父类方法”。一旦你把方法名/参数写错
    //（比如写成 speakk() 或多了个参数），那其实变成了“新方法/重载”而非重写，
    // 有 @Override 时编译器会直接报错提醒你；没有它则会悄悄编译通过、留下隐藏 bug。
    // 顺带区分：重写(override)=子类用相同签名覆盖父类方法；重载(overload)=同一个类里同名但参数不同的多个方法。
    @Override
    public String speak() { // 重写父类方法
        // super.speak() 调用父类 Animal 的实现（即使被重写也能显式访问父类版本）；
        // 常用于“在父类行为基础上扩展”，而不是完全替换。这里把父类返回的 "..." 拼进来。
        String base = super.speak();
        return name + ": Woof (父类说 " + base + ")";
    }
}
```

## 多态

- 同一个父类引用，指向不同子类对象，调用同一个方法表现不同。这是 OOP 的核心，和 C++ 虚函数一个意思。
- 区别：C++ 要显式写 `virtual` 才有动态绑定；Java 的实例方法默认就是虚的（默认动态绑定），不需要关键字。

```java
Animal a = new Dog("Rex"); // 父类引用指向子类对象
System.out.println(a.speak()); // 实际调用 Dog 的版本，输出 "Rex: Woof"
```

## 多态的实现机制（对比 C++）

- 底层和 C++ 的虚函数表（vtable）同源：每个类有一份类似 vtable 的方法表，对象头里有指针指向自己所属类的元数据。
- 调用 `a.speak()` 时，运行期按对象的**实际类型**（不是引用的声明类型）去表里查到对应实现再调用——即「动态分派 / 动态绑定」。字节码层面对应 `invokevirtual` 指令。

- 和 C++ 最大的区别：默认是否「虚」相反。
    - C++：方法默认非虚，要写 `virtual` 才进 vtable、才有多态；不写就是静态绑定（按声明类型调用）。
    - Java：实例方法默认全是虚方法，天然多态，没有也不需要 `virtual` 关键字。所以 Java 里要关心的是「哪些方法**不**参与动态分派」。

- 哪些走静态绑定（编译期定死，不查表）：`static` 方法、`private` 方法、`final` 方法、构造器，以及 `super.speak()` 这种显式指定父类版本的调用（super的调用、直接调用、不查询虚表）。
    - `final` / `private` 也方便 JIT 优化：确定不会被重写，就能放心内联，省掉查表开销。

- JIT 的额外优化：运行时若发现某个虚调用点实际只遇到一两种类型，会做「去虚化（devirtualization）+ 内联」，把查表调用优化成直接调用。所以 Java 虚调用长期运行后的开销往往比想象的小。

- 一句话：Java 多态底层就是 vtable + 动态分派，和 C++ 虚函数同源；区别在 Java 实例方法默认全虚（无需 `virtual`），而 `static`/`private`/`final`/构造器/`super.x()` 走静态绑定；此外 JIT 还能运行时去虚化提速。

## 抽象类

- `abstract class`：不能直接 new，用来作为基类定义“部分实现 + 部分留给子类实现”的模板。
- `abstract` 方法只有声明没有实现，子类必须实现，类似 C++ 的纯虚函数。

```java
public abstract class Shape {
    public abstract double area(); // 抽象方法，子类必须实现

    public String report() { // 普通方法，子类直接复用
        return "面积是 " + area();
    }
}
```

## 接口

- `interface`：定义“能力契约”，只规定有哪些方法，不关心怎么实现。类用 `implements` 实现接口。
- 一个类可以实现多个接口，这就是 Java 替代“多继承”的方式。
- 接口里的方法默认是抽象的；Java 8 起也允许写 `default` 方法（带默认实现）。

```java
public interface Drawable {
    void draw(); // 抽象方法
}

public interface Clickable {
    void onClick();
}

// 一个类可以同时实现多个接口
public class Button implements Drawable, Clickable {
    // 实现接口方法同样可以（推荐）加 @Override：在 Java 里，「实现接口方法」也算一种重写，
    // @Override 允许标注它。作用和重写父类方法时一样——让编译器校验：
    // 这个方法确实对应接口里声明的某个方法。万一方法名/参数写错（如 drow()），
    // 它就不再实现接口、且接口方法没被实现会导致编译错误，加了 @Override 能更快定位到是这里写错了。
    @Override public void draw() { System.out.println("画按钮"); }
    @Override public void onClick() { System.out.println("点击了"); }
}
```

## 抽象类 vs 接口怎么选

- 抽象类：表达“is-a”关系，且需要共享一些字段或已实现的方法。一个类只能继承一个抽象类。
- 接口：表达“能做什么”的能力，可多实现，更灵活。优先用接口做解耦。
- 经验法则：先用接口定义契约，需要复用实现细节时再考虑抽象类。

## static 成员

- `static` 修饰的字段/方法属于“类本身”，而不是某个对象，所有实例共享，用类名直接访问。
- 类似 C++ 的静态成员。`main` 方法就是 static 的，所以不用先 new 对象就能调用。

```java
public class Counter {
    private static int total = 0; // 所有对象共享这一份
    public Counter() { total++; }
    public static int getTotal() { return total; } // 用 Counter.getTotal() 调用
}
```

## record：简洁的数据类

- Java 16+ 正式支持的 `record` 用来快速定义“只装数据”的不可变类，自动生成构造器、getter、equals / hashCode / toString 等，省去样板代码。
- 很适合做后端的 DTO / 值对象。
    - DTO（Data Transfer Object，数据传输对象）：只用来「装数据、在各层之间搬运」的简单对象，本身不含业务逻辑。
    - 典型用途：接口出入参——把数据库实体转成只暴露需要字段的对象返回给前端，或把前端传来的 JSON 接成一个 DTO。好处是解耦（内部实体改动不直接影响对外结构）、只传该传的字段。
    - 值对象（value object）：由「值」而非「身份」定义、通常不可变的对象（如一个坐标点、一段金额），两个字段都相等就视为相等——`record` 自动生成的 `equals`/`hashCode` 正好契合这种语义。

```java
// 一行定义一个不可变数据类，自动有构造器和 name()/age() 访问方法
public record UserDTO(String name, int age) {}

// 使用：record 自动生成的 equals / hashCode / toString 直接可用
var u1 = new UserDTO("Alice", 20);
var u2 = new UserDTO("Alice", 20);   // 字段值和 u1 完全相同
var u3 = new UserDTO("Bob", 30);

// equals：按字段值比较（不是比引用）。所有字段相等 → true
System.out.println(u1.equals(u2)); // true（内容相同）
System.out.println(u1.equals(u3)); // false（内容不同）
System.out.println(u1 == u2);      // false（仍是两个不同对象，== 比的是引用）

// hashCode：值相等的对象 hashCode 必然相等，所以能正确用作 HashMap 的 key / HashSet 元素
System.out.println(u1.hashCode() == u2.hashCode()); // true

var set = new java.util.HashSet<UserDTO>();
set.add(u1);
System.out.println(set.contains(u2)); // true：u2 和 u1 等值，被视为已存在

// toString：自动生成可读输出，无需手写
System.out.println(u1); // UserDTO[name=Alice, age=20]
```

## equals、hashCode、toString

- `Object` 是所有类的根类，所有对象都有 `equals()`、`hashCode()`、`toString()`。
- 默认 `equals()` 比的是引用是否同一个对象；如果你希望按字段内容比较，就要重写。
- 自定义对象放进 `HashSet` 或作为 `HashMap` 的 key 时，必须让 `equals()` 和 `hashCode()` 语义一致。
    - 为什么必须一致：HashMap / HashSet 查找分两步——先用 `hashCode()` 算出落在哪个「桶（bucket）」，再在桶里用 `equals()` 逐个比对。两步必须配合才能找到对象。
    - 规则：两个对象 `equals` 为 `true`，它们的 `hashCode` 必须相等（反过来不要求，hashCode 相同允许 equals 不同，即「哈希冲突」，正常现象）。
    - 只重写 equals、不重写 hashCode 会怎样：两个「值相等」的对象可能算出不同 hashCode，被分到不同桶；查找时第一步就去错了桶，第二步的 equals 根本没机会比到 → 明明 put 进去了却 get 不到、contains 返回 false。
    - 所以两者要么都用默认（按引用），要么一起重写并基于「同一组字段」（equals() 用哪几个字段判断"相等",hashCode() 就该用同样那几个字段来计算哈希值——两个方法依据的字段集合要完全相同），保证语义一致。`record` 正是自动做到了这点。
- `record` 会自动生成这些方法，所以很适合做后端 DTO、值对象、Map key。

## 没有析构函数

- Java 没有 C++ 那种确定时机执行的析构函数，不要指望对象被 GC 时自动释放文件、连接、Socket。
- 资源释放交给 `try-with-resources`，具体放在[异常处理](异常处理.md)和[IO与文件](IO与文件.md)里看。

```java
// try (...) 括号里声明的资源，无论正常结束还是抛异常，都会在块结束时自动调用 close()。
// 前提：资源类型实现了 AutoCloseable 接口（文件、流、连接等大多都实现了）。
try (var reader = new java.io.BufferedReader(new java.io.FileReader("data.txt"))) {
    System.out.println(reader.readLine());
} catch (java.io.IOException e) {
    e.printStackTrace();
}
// 出了这个块，reader 已被自动关闭——这就是 Java 版的“确定时机释放资源”，
// 用语法块的结束代替 C++ 析构函数的作用，不依赖 GC。
```
