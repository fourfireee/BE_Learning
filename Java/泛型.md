# Java 泛型

- 泛型让你写出“类型参数化”的代码：同一套逻辑适配不同类型，同时保留编译期类型检查。
- 你熟悉 C++ 模板，所以这篇重点讲 Java 泛型和 C++ 模板的本质区别。

## 基本用法

- 用尖括号 `<T>` 声明类型参数，T 是占位符，使用时再填具体类型。

```java
// 一个能装任意类型的简单容器
public class Box<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}
```

```java
Box<String> b = new Box<>();  // 指定 T = String
b.set("hello");
String s = b.get();           // 编译器知道返回的是 String，不用强转
```

- 泛型方法：在返回类型前用 `<T>` 声明：

```java
// <T> 声明类型参数，参数和返回值都用 T
public static <T> T firstOf(java.util.List<T> list) {
    return list.get(0);
}
```

## 和 C++ 模板的本质区别：类型擦除

- 这是最关键的概念。C++ 模板是“编译期实例化”：`vector<int>` 和 `vector<string>` 会各自生成一份独立的机器码。
- Java 泛型是“类型擦除 (type erasure)”：泛型信息只在编译期用于类型检查，编译后类型参数会被擦掉，运行时统一退化成 `Object`（或边界类型）。
- 也就是说，运行时只有一个 `Box` 类，`Box<String>` 和 `Box<Integer>` 在运行时是同一个类。

由此带来几个 C++ 没有的限制：

- 不能 `new T()`：运行时不知道 T 是什么，无法创建实例。
- 不能 `new T[]`：同理不能创建泛型数组。
- 运行时拿不到具体类型：`list instanceof List<String>` 这种写法不允许。
- 基本类型不能直接当类型参数：必须用包装类，写 `List<Integer>` 而不是 `List<int>`。

```java
// C++ 里可以 T t = T(); Java 里不行，因为运行时 T 已被擦除
// 需要的话要额外传入 Class<T> 或工厂方法来绕过
```

## 通配符与边界

- `extends` 设上界：`T extends Number` 表示 T 必须是 Number 或其子类，这样就能调用 Number 的方法。

```java
// 限定 T 是 Number 的子类，才能调用 doubleValue()
public static <T extends Number> double sum(java.util.List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue();
    return total;
}
```

- 通配符 `?` 表示“某种未知类型”，常见于方法参数：
- `List<? extends Number>`：读取场景，能读出 Number，但不能往里加（不知道具体子类）。
- `List<? super Integer>`：写入场景，能塞 Integer，但读出来只能当 Object。
- 记忆口诀 PECS：Producer Extends, Consumer Super（生产者用 extends，消费者用 super）。

## 泛型是不变的

- `Integer` 是 `Number` 的子类，但 `List<Integer>` 不是 `List<Number>` 的子类。
- 原因很直观：如果允许把 `List<Integer>` 当成 `List<Number>`，那就可以往里面塞 `Double`，破坏原本只装 Integer 的约束。

```java
java.util.List<Integer> ints = java.util.List.of(1, 2, 3);
// java.util.List<Number> nums = ints; // 编译错误

// 只读 Number 时，用 extends 通配符
java.util.List<? extends Number> nums = ints;
Number n = nums.get(0);
```

## 实战提示

- 日常后端开发，绝大多数时候你只是在“使用”泛型容器（`List<User>`、`Map<String, User>`），自己定义泛型类的机会不多。
- API 入参里如果只是读取元素，可以考虑 `List<? extends BaseType>`；如果要写入元素，才考虑 `List<? super SubType>`。普通业务代码不用过度设计。
- 看到编译期 `unchecked` 警告，通常是因为和老代码、类型擦除打交道，理解擦除机制就能看懂原因。
- 一句话对比：C++ 模板像“代码生成器”，每个类型一份代码、性能零开销；Java 泛型像“编译期类型检查的语法糖”，运行时只有一份代码。
