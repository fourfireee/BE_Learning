# Java 构建与依赖

- 真实项目不会手敲 `javac`，而是用构建工具管理依赖、编译、测试、打包。Java 生态主要是 Maven 和 Gradle。
- 你可以把它们类比成 C++ 世界的 CMake + 包管理器（vcpkg/conan）的合体。

## 为什么需要构建工具

- 自动下载依赖：从中央仓库拉第三方库（jar 包），自动处理“依赖的依赖”。
- 统一编译流程：约定项目结构，一条命令完成编译、测试、打包。
- 可复现：所有人用同一份配置和版本，避免“在我机器上能跑”。
- 对比 C++：C++ 长期缺统一的包管理，要手动找库、配头文件和链接；Java 这块成熟很多。

## Maven

- 最经典的构建工具，用 XML 文件 `pom.xml` 描述项目。
- `pom` = Project Object Model。约定的标准目录结构：

```txt
my-app/
  pom.xml                       # 项目配置：依赖、插件、打包方式
  src/
    main/
      java/                     # 业务代码
      resources/                # 配置文件等资源
    test/
      java/                     # 测试代码
```

- 最小 `pom.xml`：

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 https://maven.apache.org/xsd/maven-4.0.0.xsd">
  <modelVersion>4.0.0</modelVersion>
  <!-- 坐标：唯一标识一个项目/库，三段式 groupId:artifactId:version -->
  <groupId>com.example</groupId>
  <artifactId>my-app</artifactId>
  <version>1.0.0</version>

  <properties>
    <!-- 用固定 Java 版本编译，避免 Maven 默认源码版本过旧 -->
    <maven.compiler.release>17</maven.compiler.release>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  </properties>

  <dependencies>
    <!-- 声明一个依赖，Maven 会自动从中央仓库下载这个 jar 及其依赖 -->
    <dependency>
      <groupId>com.google.guava</groupId>
      <artifactId>guava</artifactId>
      <version>33.0.0-jre</version>
    </dependency>
  </dependencies>
</project>
```

- 关键概念 `坐标 (coordinates)`：`groupId:artifactId:version` 三段唯一定位一个库，类似 npm 的包名+版本。

常用命令：

```bash
mvn compile      # 编译
mvn test         # 运行测试
mvn package      # 打包成 jar（产物在 target/ 目录）
mvn clean        # 清理构建产物
mvn clean install # 清理 + 构建 + 安装到本地仓库
```

## Gradle

- 更现代、更灵活，用 Groovy 或 Kotlin 脚本写配置（`build.gradle` / `build.gradle.kts`），比 XML 简洁。
- 构建速度通常更快（增量构建、缓存好）。Android 和很多新项目首选。

- 最小 `build.gradle`：

```groovy
plugins {
    id 'java' // 启用 Java 构建能力
}

repositories {
    mavenCentral() // 从 Maven 中央仓库拉依赖（Gradle 复用同一套仓库）
}

dependencies {
    // 同样的坐标，写法更紧凑
    implementation 'com.google.guava:guava:33.0.0-jre'
}
```

常用命令：

```bash
./gradlew build   # 编译 + 测试 + 打包（gradlew 是项目自带的包装脚本，保证版本一致）
./gradlew test    # 运行测试
./gradlew clean   # 清理
```

## 依赖范围（scope）

- 依赖不是都一样用，常见范围：
- 编译+运行都需要：Maven 的 `compile`（默认）/ Gradle 的 `implementation`。
- 只测试用：Maven 的 `test` / Gradle 的 `testImplementation`，如 JUnit。
- 只编译用、运行时由环境提供：Maven 的 `provided`。

## 传递依赖与版本冲突

- `传递依赖`：你依赖 A，A 又依赖 B，构建工具会自动把 B 也拉进来，不用你手动声明。
- `版本冲突`：不同库依赖了同一个库的不同版本，工具会按规则选一个（Maven 选“最近”的，Gradle 默认选最高版本）。冲突时可用命令查看依赖树：

```bash
mvn dependency:tree       # Maven 查看依赖树
./gradlew dependencies    # Gradle 查看依赖树
```

## Maven Wrapper / Gradle Wrapper

- 团队项目里常见 `mvnw`、`gradlew` 这类 wrapper 脚本，用来固定构建工具版本。
- 好处是新人不用先全局安装同版本 Maven/Gradle，直接运行项目自带脚本即可。
- 类比 C++ 项目把工具链版本写进配置，避免每个人机器上的构建工具行为不一致。

## jar 包是什么

- `jar` = Java ARchive，本质是个 zip，把编译后的 `.class` 和资源打包在一起，是 Java 的分发和依赖单位。
- 可执行 jar 里有个清单指明入口类，可以直接 `java -jar app.jar` 运行。

## 怎么选

- 新项目、追求灵活和速度：Gradle。
- 老项目、企业环境、求稳定和资料多：Maven。
- 两者用的是同一套中央仓库和坐标体系，概念互通，学会一个另一个很快上手。

## 可运行示例

- 这是一个最小的 Maven 项目，演示“声明依赖 → 自动下载 → 在代码里使用第三方库 → 打包成可执行 jar”。
- 项目结构：
    - [`examples/maven-demo/pom.xml`](examples/maven-demo/pom.xml)：项目配置，声明了一个第三方依赖 commons-lang3。
    - [`examples/maven-demo/src/main/java/com/example/App.java`](examples/maven-demo/src/main/java/com/example/App.java)：入口类，调用第三方库的工具方法。
- 前置：需要先装 Maven（`brew install maven`），它会自动联网下载依赖。
- 运行：

```bash
cd Java/examples/maven-demo

# 方式一：编译后直接用 Maven 运行
mvn compile
mvn exec:java -Dexec.mainClass=com.example.App

# 方式二：打成带依赖的可执行 jar 再运行
mvn package
java -jar target/maven-demo-1.0.0.jar
```

- 预期输出：把 `  java maven  ` 处理成 `Java maven`，并判断空白字符串。
